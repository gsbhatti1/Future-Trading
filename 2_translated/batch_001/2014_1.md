**Principle of the Strategy**

The principle behind this strategy is extremely simple, essentially acting as a quasi-high-frequency market-making strategy. After reading about it, one might think: This can make money? Back in 2014 when bitcoin trading robots were first gaining traction, creating profitable strategies was incredibly easy.

Like all high-frequency strategies, this one is based on the orderbook. Below is an example of a typical order distribution for a Bitcoin exchange:
![](https://dn-filebox.qbox.me/0d8ec18c831404d3d1c19e17299c78017abcfd48.png)

The left side shows buy orders, with different prices and quantities. The right side shows sell orders. If someone wants to buy Bitcoin without placing an order, they would have to take the bid (eat a bid), which could quickly fulfill multiple sell orders on the ask side and impact the price significantly. However, such impacts do not persist; others are likely to take the offer (eat an offer) at any time, causing the price to revert in a very short period. The reverse is also true for someone selling Bitcoin.

Using the example from the image, if you wanted to buy 5 bitcoins directly, the price would rise to 10377. If another party were to sell 5 bitcoins directly at that moment, the price could drop to 10348. The space between these two prices (28.98) is where the profit lies. The strategy places an order slightly below 10377, say at 10376.99, and simultaneously buys above 10348, for example, at 10348.01. If this scenario occurs, you would obviously make a difference of 28.98 between the two prices. While it doesn’t always work perfectly, given the probabilistic nature, there is an incredibly high chance of making money.

To explain with current strategy parameters: The parameter settings cannot be used now; they are provided for explanatory purposes only. It would look for a sell order accumulation of 8 bitcoins to determine the price at 10377. At this point, the selling price would be that price minus 0.01 (the exact amount can vary randomly). Similarly, it would find a buy order accumulation of 8 bitcoins, determined as 10348, and set the buying price to 10348.01. The difference between these two prices is 28.98, which exceeds the pre-set spread (1.5), so orders are placed at those levels; if the difference is less than 1.5, a price close to that of the current market would be chosen for placing an order and waiting for execution (a better approach might be to continue searching deeper).

**Further Explanations**

1. What if there’s no money or coins?
In my case, this scenario was common when I had limited funds; often, orders were placed on only one side but it wasn’t a big problem. Actually, balance and coin logic could have been added, but the process would likely incur losses given that each execution is a matter of probability. So, I chose to wait for a single-side order to complete, even though this wasted opportunities from the other side.
2. How is position management handled?
Initially, trades were always fully invested; later, different groups were used based on various parameters, but not all orders would be filled in one go.
3. Is there any stop-loss mechanism?
The strategy has a complete order placement logic; I believe it doesn’t need a stop-loss (open to discussion), and the chance of making money is so high that selling at a loss seems regrettable.
4. How can this be adjusted for profit from coins instead of profits in fiat currency?
With symmetric parameters, i.e., an accumulation of 8 sell orders on top and 8 buy orders below, slightly unbalancing it—say by increasing the upward sell order count to 15—makes it more difficult to sell coins, with a higher probability of buying them back at lower prices. This would result in profit from coins instead of fiat currency. Conversely, this approach results in profits in fiat currency.

function CancelPendingOrders() {
    var orders = _C(exchange.GetOrders);
    for (var j = 0; j < orders.length; j++) {
        exchange.CancelOrder(orders[j].Id, orders[j]);
    }
}

// Function to calculate the price at which an order should be placed
function GetPrice(Type, depth) {
    var amountBids = 0;
    var amountAsks = 0;

    // Calculating buy price by finding a cumulative bid depth equal to or exceeding pre-set value
    if (Type == "Buy") {
        for (var i = 0; i < 20; i++) {
            amountBids += depth.Bids[i].Amount;
            if (amountBids > floatamountbuy) {
                // Add a small margin to ensure the order is prioritized
                return depth.Bids[i].Price + 0.01;
            }
        }
    }

    // Calculating sell price by finding a cumulative ask depth equal to or exceeding pre-set value
    if (Type == "Sell") {
        for (var j = 0; j < 20; j++) {
            amountAsks += depth.Asks[j].Amount;
            if (amountAsks > floatamountsell) {
                return depth.Asks[j].Price - 0.01;
            }
        }
    }

    // If no sufficient order depth is found, fall back to the first ask price
    return depth.Asks[0].Price;
}

function onTick() {
    var depth = _C(exchange.GetDepth);
    var buyPrice = GetPrice("Buy", depth);
    var sellPrice = GetPrice("Sell", depth);

    if ((sellPrice - buyPrice) <= diffprice) {

        sellPrice += 10;
